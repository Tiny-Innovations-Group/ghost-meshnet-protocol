## 1. The 433MHz Control Plane: The 104-Byte Beacon
> **Design Constraint:** Absolute 32-bit and 64-bit CPU memory alignment to prevent wasted clock cycles during SPI hardware transfers.

The 433MHz channel acts as the decentralized directory server and air-traffic controller. To optimize Time-on-Air (ToA) and guarantee perfect memory alignment on the RP2040 microcontroller, the Discovery Beacon is rigidly constrained to exactly **104 bytes**. 

By compressing the network timestamp from *seconds* to a 24-bit *minutes* counter (providing ~32 years of rollover protection), we merge the timestamp, power telemetry, and control flags into a single, perfectly aligned 32-bit (4-byte) unified header.

* **Mathematical Alignment:** 104 bytes perfectly divides into 26 cycles on a 32-bit CPU, and 13 cycles on a 64-bit CPU. Zero padding is required. Zero clock cycles are wasted.



### The 104-Byte Profile Breakdown
* **Unified Header:** 4 bytes *(Control Type, Power State, 24-bit Epoch Minutes)*
* **Node ID:** 4 bytes
* **Public Identity Key:** 32 bytes *(Curve25519)*
* **Hardware Signature:** 64 bytes *(Ed25519 Hardware-backed PUF/Enclave signature)*

## 2. The 868MHz Data Plane: The 124-Byte Onion
> **Design Constraint:** Maximize the 255-byte LoRa hardware FIFO buffer to achieve a "Dual-Relay Burst," doubling the legal network capacity within the 1% ETSI/Ofcom duty cycle.

The 868MHz band is completely silent until coordinated by the 433MHz Control Plane. Once cleared, it fires a "Dual-Burst" of two independent packets simultaneously. To achieve this, the absolute maximum size of an Onion packet must be exactly **124 bytes** (124 * 2 = 248 bytes, perfectly filling the buffer and aligning to 32-bit memory).

To achieve this without sacrificing payload capacity, the metadata is aggressively bit-packed into a single **8-bit Control Byte**, freeing up crucial space for the Static Huffman compressed text payload (now 77 bytes, capable of holding ~128 raw characters).

### The 124-Byte Profile Breakdown
**Layer 2 & Layer 1: Outer Routing Shells (12 bytes)**
* Next Hop ID: 4 bytes
* Truncated Auth Tag (MAC): 8 bytes 

**Layer 3: The Inner Core (112 bytes)**
* Sender Node ID: 4 bytes
* Full Auth Tag (MAC): 16 bytes
* Message ID / Implicit Nonce: 2 bytes
* **Control Byte:** 1 byte *(Payload Type, Priority, ACK, TTL)*
* Compressed Text Payload: 77 bytes

You are absolutely right. This is the difference between writing documentation for *humans* and writing a specification for *compilers*. 

If we just use the `doc` tag, the Kaitai compiler ignores it when generating the C++ code. The RP2040 firmware developer would just get a raw integer and have to manually write `if (payload_type == 2)` in their code, which is dangerous and prone to typos. 

By hardcoding these into the `enums` block in Kaitai, the compiler will generate **strongly-typed C++ enum classes**. This means if a developer tries to pass an invalid priority level, the C++ compiler will literally refuse to build the firmware. That is how you enforce protocol safety at the silicon level.
